# 文件拖出交互方案

> 目标:让 Folio 里的图片可以像系统看图/资源管理器一样拖到文件夹复制,也可以拖到网页上传控件。拖出的是原始文件路径,不是 `gv-img://`、base64 或临时二进制。

## 交互原则

1. **单图 Fit 状态:从图片本身拖出当前文件**
   - 图片完整适应画布时,用户拖图片即表示“拿起这张文件”。
   - 可拖到资源管理器/Finder 复制,也可拖到浏览器网页上传。
   - 这是主入口,对齐 Windows 看图的心智。

2. **放大状态:拖图片优先平移**
   - 只要图片处于非 Fit 的缩放状态,拖动图片仍然是平移画布。
   - 避免“想拖动画面却把文件拖走”的误触。
   - 该状态下如需拖出文件,使用队列侧栏入口。

3. **队列侧栏:整行可拖出文件**
   - 侧栏行永远表示文件实体,拖出行为不受 Viewer 缩放状态影响。
   - 未来多选后,侧栏拖拽自然扩展为拖出多张文件。

4. **多图模式:拖鼠标所在格子的图片**
   - 每个格子对应自己的文件,拖出时应使用鼠标按下所在格子的路径,不能误用当前焦点之外的文件。
   - 如果该格未来支持独立放大/平移,仍按“放大优先平移”的规则处理。

## 架构约束

- Renderer 不读取文件、不拼接文件二进制、不通过 IPC 传图片内容。
- Renderer 只把当前队列项的真实文件路径交给 preload。
- Preload 只转发拖拽请求。
- Main 进程调用 Electron 原生 `webContents.startDrag`,由系统拖放机制把文件交给资源管理器、Finder 或浏览器。
- `gv-img://` 只用于应用内显示,不得作为外部拖放数据。

## 分阶段落地

### Phase 1:Canvas Fit 图片拖出

- 范围:Canvas 单图视图下,当前图片可原生渲染且 Viewer 处于 Fit;包括普通单图和多图临时展开后的焦点单图。
- 行为:在图片 `dragstart` 时阻止 DOM 默认拖拽,通过 IPC 请求 main 开始系统文件拖拽。
- 不包含:侧栏拖出、多图格子拖出、批量拖出。

### Phase 2:队列侧栏拖出

- 范围:队列行整行可拖。
- 行为:拖单行时拖出该行图片;未来存在多选时,拖出选中集合。
- 注意:拖拽排序如果未来加入,需用拖拽起点和修饰键区分。

### Phase 3:多图格子拖出

- 范围:双图/三图/四宫格。
- 行为:拖哪个格子就拖出哪个格子的文件;单击/拖动可先设为焦点,但拖出路径以指针所在格子为准。

### Phase 4:多文件拖出

- 范围:侧栏多选、当前组拖出。
- 行为:main 使用 Electron 多文件拖拽能力;批量操作前不需要确认,因为拖放目标由用户在系统层面决定。

## 当前状态

- 已实现 Phase 1。
- 后续优先级建议:Phase 2 高于 Phase 3,因为侧栏可作为所有缩放状态下的稳定文件入口。
